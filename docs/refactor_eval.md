# 最简单的case(单个仓库)
1. 用户在web界面输入 repo url 和 committer name
2. 系统拉取该 repo 的 commit 历史, commit中包括所有的diff 和 message等消息.
3. 从repo开始的时间算起, 每两周作为一个周期, 每10个commit 作为一个评估节点, 评估该节点(所有10个commits)的代码质量. 如果某个周期内不满10个commits, 则跳过该周期的评估, 累积到下一个周期.直到满足10个commits 以及以上的数量.
举例: 
  1月1日到1月14日内有8个commits, 则跳过该周期; 
  1月15日到1月28日内有1个commits, 则累积前一个周期的8个commits, 一共9个commits, 仍然不满足10个commits, 则继续累积到下一个周期, 直到满足10个commits数量;
  1月29日到2月11日内有5个commits, 则累积前两个周期的9个commits, 一共14个commits, 则评估这14个commits;
  2月12日到2月25日内有12个commits, 则评估这12个commits.
4. 一般分数呈现递增形势, 除非出现明显的负面证据. 比如: 第一个节点多维评估分数为:[20, 10, 0, 60], 第二个评估节点分数一般应不低于第一个节点, 例如:[30, 10, 00, 70]. 如果第二个节点分数低于第一个节点, 则需要有明显的负面证据支撑, 比如代码质量变差, 出现bug等.所以第二个评估节点需要依赖第一个节点的评估结果, 进行对比分析.
5. 系统生成每个评估节点的评估分数和总结, 存储.
6. 界面上展示每个评估节点的分数的时间序列变化趋势图表(折线图).

# 多个仓库case
分别对每个仓库进行上述单个仓库的评估过程, 生成每个仓库的评估结果.

# 一个仓库多个用户名属于一个人
在group设置界面允许用户输入多个用户名(以英文逗号分隔), 系统会将这些用户名视为同一个人, 对该人的所有用户名进行提交的commits, 2-week periods and 10-commit minimums的评估, 生成该人的评估结果.
