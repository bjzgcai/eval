# OSS Audit 评分标准

## 概述

OSS Audit 采用多维度评分体系，每个维度都有明确的评分标准和权重。评分基于实际工具扫描结果、文件检测和配置分析，确保评估结果的客观性和准确性。

## 评分体系

### 总体评分范围
- **0-100分**：每个维度的满分
- **等级划分**：
  - 90-100分：优秀 (Excellent)
  - 80-89分：良好 (Good)
  - 70-79分：一般 (Fair)
  - 50-69分：较差 (Poor)
  - 0-49分：极差 (Critical)

### 项目成熟度级别
- **企业级就绪 (Enterprise Ready)**：平均分 ≥ 80分
- **生产就绪 (Production Ready)**：平均分 ≥ 70分
- **开发就绪 (Development Ready)**：平均分 ≥ 50分
- **实验性 (Experimental)**：平均分 < 50分

## 各维度评分标准

### 维度1：代码结构与可维护性

**评分权重**：
- Pylint代码质量检查：50%
- SonarQube静态分析：30%
- 项目结构完整性：20%

**详细标准**：
- **Pylint分数**：
  - 90-100分：无问题或极少问题
  - 80-89分：少量问题（≤5个）
  - 70-79分：中等问题（6-10个）
  - 60-69分：较多问题（11-20个）
  - <60分：大量问题（>20个）

- **SonarQube分析**：
  - 85-100分：高质量代码，符合最佳实践
  - 70-84分：良好质量，基本符合规范
  - 50-69分：一般质量，需要改进
  - <50分：质量较差，需要重构

- **项目结构**：
  - 包含README.md、LICENSE、requirements.txt等基础文件
  - 合理的目录结构
  - 代码组织清晰

### 维度2：测试覆盖与质量保障

**评分权重**：
- 测试覆盖率：60%
- 测试文件数量：20%
- 测试执行成功率：20%

**详细标准**：
- **测试覆盖率**：
  - 90-100%：优秀覆盖率
  - 80-89%：良好覆盖率
  - 70-79%：一般覆盖率
  - 50-69%：较低覆盖率
  - <50%：覆盖率不足

- **测试文件数量**：
  - ≥10个：测试充分
  - 5-9个：测试基本充分
  - 2-4个：测试较少
  - <2个：测试不足

### 维度3：构建与工程可重复性

**评分权重**：
- Docker支持：40%
- 构建脚本：30%
- 依赖管理：30%

**详细标准**：
- **Docker支持**：
  - 有Dockerfile且构建成功：40分
  - 有Dockerfile但构建失败：20分
  - 无Dockerfile：0分

- **构建脚本**：
  - 有Makefile/构建脚本且可执行：30分
  - 有构建脚本但执行失败：15分
  - 无构建脚本：0分

- **依赖管理**：
  - 有requirements.txt且格式正确：30分
  - 有依赖文件但格式不规范：15分
  - 无依赖管理文件：0分

### 维度4：依赖与许可证合规

**评分权重**：
- 许可证文件：40%
- 依赖管理：40%
- 许可证合规性：20%

**详细标准**：
- **许可证文件**：
  - 有标准开源许可证：40分
  - 有许可证但格式不规范：20分
  - 无许可证文件：0分

- **依赖管理**：
  - 有requirements.txt且依赖明确：40分
  - 有依赖文件但信息不完整：20分
  - 无依赖管理：0分

### 维度5：安全性与敏感信息防护

**评分权重**：
- 安全扫描结果：50%
- 敏感信息检查：30%
- 安全配置：20%

**详细标准**：
- **安全扫描**：
  - 无安全问题：50分
  - 1-5个低风险问题：40分
  - 6-10个问题：30分
  - >10个问题：20分

- **敏感信息**：
  - 无敏感信息泄露：30分
  - 少量敏感信息：15分
  - 大量敏感信息：0分

### 维度6：CI/CD 自动化保障

**评分权重**：
- CI/CD配置：60%
- 自动化测试：40%

**详细标准**：
- **CI/CD配置**：
  - 有GitHub Actions/GitLab CI等：60分
  - 有CI配置但未启用：30分
  - 无CI/CD配置：0分

- **自动化测试**：
  - CI中包含自动化测试：40分
  - 有测试但未自动化：20分
  - 无测试：0分

### 维度7：使用文档与复现性

**评分权重**：
- README质量：40%
- 文档完整性：40%
- 文档可读性：20%

**详细标准**：
- **README质量**：
  - 内容完整、结构清晰：40分
  - 内容基本完整：25分
  - 内容简单：10分
  - 无README：0分

- **文档完整性**：
  - 包含安装、使用、贡献指南：40分
  - 包含基本使用说明：25分
  - 文档不完整：10分

### 维度8-14：其他维度

**评分标准**：
- 基于文件存在性和配置完整性
- 每个维度满分100分
- 根据检测到的文件和配置质量评分

## AI分析增强

### 智能评分调整
- **数据完整性检查**：根据扫描数据的完整性调整评分
- **异常检测**：识别评分异常并进行修正
- **置信度计算**：为每个评分提供置信度指标

### 改进建议生成
- **个性化建议**：基于项目具体情况生成改进建议
- **优先级排序**：按影响程度排序改进建议
- **时间估算**：为每个改进建议提供时间估算

### 成熟度分析
- **整体评估**：基于所有维度综合评估项目成熟度
- **改进路线图**：生成分阶段的改进计划
- **对标分析**：与同类型项目进行对比分析

## 评分算法

### 基础评分公式
```
维度得分 = Σ(子项得分 × 权重) × 调整系数
```

### 调整系数
- **数据完整性**：0.8-1.2
- **工具可用性**：0.9-1.1
- **项目规模**：0.95-1.05

### 最终评分
```
项目总分 = Σ(维度得分) / 维度数量
成熟度等级 = 根据总分确定对应等级
```

## 评分更新机制

### 定期更新
- 每季度更新评分标准
- 根据行业最佳实践调整权重
- 增加新的评估维度

### 反馈机制
- 收集用户反馈
- 分析评分准确性
- 持续优化算法

## 使用说明

1. **客观评估**：评分基于实际检测结果，确保客观性
2. **全面分析**：涵盖14个关键维度，提供全面评估
3. **可操作建议**：提供具体的改进建议和路线图
4. **持续改进**：支持定期重新评估，跟踪改进进度

通过这套评分标准，OSS Audit能够为开源项目提供科学、客观、可操作的成熟度评估。
